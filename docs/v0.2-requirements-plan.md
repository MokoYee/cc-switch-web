# CC Switch Web v0.2 需求分析与功能计划

## 0. 当前状态快照（2026-03-30）

本轮已经完成以下 `P0` 交付：

- 已新增真实 CLI 兼容性验收矩阵文档与执行脚本。
- 已补齐 ProviderType 差异化健康检查、恢复验证阈值和恢复中提示。
- 已为 `codex`、`claude-code` 提供 `environment-override` 初版接管模式。
- 已补 `ccsw daemon service logs` / `follow` 与更明确的 doctor 推荐动作。

当前 `v0.2` 的下一优先级已经从“补齐 P0”转向“收口运维闭环、实机验收记录和文档同步”。

## 1. 文档目的

本文件用于解决两个问题：

- 当前 `v0.1.x` 已开源后，下一阶段到底该优先补什么。
- 后续协作与新会话进入仓库时，如何避免继续依赖已部分过时的早期路线图判断。

本文件不替代 `.agent-context/cc-switch-gap-roadmap.md`，而是基于当前真实实现状态，对 `v0.2` 做一次更贴近执行层的收口。

## 2. 当前版本判断

当前仓库已经具备以下成立事实：

- 已完成 npm 包首版公开发布，主包名为 `cc-switch-web`，CLI 主命令为 `ccsw`。
- 项目主形态已经稳定为 `daemon-first`。
- 单端口 daemon、内置 `/ui/` 控制台、SQLite 持久化、配置快照、导入导出、基础 CLI 已具备最小可用闭环。
- `OpenAI-compatible` 代理直通已可用。
- Anthropic 到 OpenAI-compatible 的最小桥接、文本 SSE、工具调用、工具流、图片请求兼容已经落地。
- `FailoverChain`、运行态健康状态、最近请求日志、审计事件、Prometheus 指标、`systemd --user` 管理链路均已具备基础能力。
- 宿主机侧对 `Codex`、`Claude Code`、`Gemini CLI`、`OpenCode`、`OpenClaw` 的发现、接管、MCP Host Sync、Prompt Host Sync 已经不是纯骨架。

结论：

- 当前阶段不再是“能不能跑起来”。
- 当前阶段的主矛盾已经变成“真实 CLI 场景是否足够稳定”和“Linux 宿主机长期运行是否足够可靠”。

## 3. 当前主要缺口

结合现有实现和首版开源后的产品目标，当前最关键的缺口不是页面数量，而是以下五类：

### 3.1 真实 CLI 兼容性已有矩阵，但还缺持续实机结论

- 兼容性矩阵文档和执行脚本已经落地，但还缺按版本、按 CLI 实际记录的验收结果沉淀。
- `Codex`、`Claude Code` 需要继续补实机结论，避免只停留在脚本层面。
- `Gemini CLI` 当前已明确标记阻塞，阻塞原因需要持续跟踪，而不是模糊地写成“待验证”。

### 3.2 健康检查与故障转移已进入稳定化阶段

- 当前已经补上 ProviderType 差异化探测与恢复验证阈值。
- 下一步重点不再是“有没有探活”，而是“长期运行下是否足够可信、是否便于排障”。
- 需要继续观察长时间抖动、连续恢复失败和复杂 failover 链下的行为。

### 3.3 宿主机接管已有双模式，但还缺更完整运行闭环

- `file-rewrite` 与 `environment-override` 双模式已经具备。
- `environment-override` 当前采用受管脚本 + 显式 `source` 的安全方案，不会自动修改 shell rc。
- `Gemini CLI` 仍不能承诺 takeover 可用，根因是代理主链路暂未提供 Gemini API / Gateway 适配。

### 3.4 运维闭环还不够完整

- 当前已有 `systemd --user` 的 install/status/doctor。
- 但缺少面向真实排障的服务日志查看、跟随和一键诊断闭环。
- 这会导致开源用户仍然需要手工使用 `journalctl`、`curl`、`systemctl` 拼接排障流程。

### 3.5 路线图和当前实现仍有一定脱节

- 现有 `.agent-context/cc-switch-gap-roadmap.md` 仍有参考价值。
- 但其中的成熟度估计、部分“待补模型”、当前优先级描述，已经落后于当前真实实现。
- 不更新会影响后续会话连续性，也会干扰功能优先级判断。

## 4. v0.2 版本目标

`v0.2` 不追求“一次性补齐所有厂商和所有高级能力”，而是聚焦下面这个目标：

> 让 Linux 宿主机用户可以把至少一类真实 AI CLI 稳定接到 CC Switch Web，并在常见故障场景下完成观测、切换、恢复和排障。

### 4.1 v0.2 应该达到的标准

- 用户能把 `Codex` 或 `Claude Code` 接到本地代理并稳定使用。
- 主 Provider 故障时，系统能进行可信的自动切换或明确降级。
- 用户能通过控制台或 CLI 明确看到当前绑定、运行状态、请求结果和最近故障。
- 用户能在 `systemd --user` 模式下完成安装、验证、查日志和回滚。
- README 和内部交接文档对当前产品边界描述一致。

### 4.2 v0.2 不做的事情

- 不追求一次性补齐所有协议边界。
- 不追求一次性支持所有第三方 CLI 的深度接管。
- 不优先做大量视觉层优化。
- 不引入与当前 Node.js 单机架构不匹配的复杂分布式设计。

## 5. v0.2 优先级规划

### P0：已完成

#### P0-1 真实 CLI 兼容性验收矩阵

当前状态：

- 已建立以真实 CLI 为中心的验收基线，入口为 `docs/cli-compatibility-matrix.md` 和 `npm run acceptance:cli:matrix`。

范围：

- `Codex`
- `Claude Code`
- `Gemini CLI`

最小验收维度：

- 非流式文本请求
- 流式文本请求
- 工具调用
- 图片输入
- 上游超时
- 上游 429 / 5xx
- failover 后行为
- 错误信息可读性

已交付：

- 一份可执行的兼容性矩阵文档
- 一组可复现的手工验证脚本或 CLI 操作步骤
- 支持按 app / 场景筛选的矩阵执行脚本

仍需继续：

- 按版本沉淀真实 `Codex` / `Claude Code` 验收记录
- 持续保留 `Gemini CLI` 的阻塞原因和解锁条件

#### P0-2 健康检查与 failover 韧性增强

当前状态：

- 已把故障转移从“最小可用”推进到“可托管试用”阶段。

范围：

- 区分不同 ProviderType 的探测 URL 与判定逻辑
- 更细的失败分类
- 半开恢复与恢复窗口细化
- 并发 probe 去重与状态机保护
- failover 事件与恢复事件的更清晰展示

已落实：

- 主 Provider 失效时，重复失败不会短时间内持续击穿同一节点
- 恢复后不会因为一次偶发成功就过早判定全量恢复
- 控制台能解释“为什么切换”“为什么仍未恢复”

仍需继续：

- 长时间 soak 场景下继续验证恢复窗口阈值是否合适
- 补更明确的服务日志与诊断闭环，降低排障成本

#### P0-3 环境变量接管模式初版

当前状态：

- 已在文件写回之外提供更安全、更易回滚的宿主机接管方案。

当前范围：

- 为 `Codex` / `Claude Code` 提供 env takeover 方案
- 生成建议导出脚本或 shell snippet
- 允许控制台预览变更影响
- 保留文件写回模式，但明确两种模式的差异与风险

当前限制：

- `Gemini CLI` 暂不提供 env takeover apply，阻塞点是 Gemini API / Gateway 主链路适配未完成

已满足：

- 用户不改原配置文件，也能通过环境变量把 CLI 指向本地代理
- 停用接管时，宿主机可以回到原状

### P1：v0.2 当前主战场

#### P1-1 `systemd --user` 运维闭环

当前状态：

- 已提供 `ccsw daemon service logs [--lines <n>]`
- 已提供 `ccsw daemon service follow [--lines <n>]`
- 已支持 `--since` / `--until` / `--grep` 基础过滤
- `Service Doctor` 推荐动作已改为可直接执行的 CLI 命令
- runbook 已补标准排障顺序

范围：

- `ccsw daemon service logs`
- `ccsw daemon service follow`
- 更明确的 doctor 输出
- runbook 中增加标准排障路径

目标：

- 开源用户不需要自己拼装 `systemctl` + `journalctl` + `curl` 的诊断流程。

仍需继续：

- 视真实用户反馈决定是否补时间范围、grep 过滤或更细粒度日志导出

#### P1-2 路线图与交接文档收口

范围：

- 更新 `.agent-context/cc-switch-gap-roadmap.md`
- 更新 `.agent-context/new-project-handoff.md`
- 更新 `.agent-context/new-session-bootstrap-prompt.md`
- 视需要补 README 中的“下一步能力边界”

目标：

- 后续会话直接以 `v0.2` 计划作为真实基线。

#### P1-3 真实 Provider 编辑和模型映射增强

范围：

- 按 ProviderType 展开更明确的字段校验
- 模型映射与默认值策略
- 更清晰的 credential 录入与提示

目标：

- 降低首次配置门槛，减少“表单能保存但运行失败”的情况。

### P2：稳定性与开源完成度增强

范围：

- 导入导出兼容性补强
- 敏感配置导出确认
- 更完整的审计与请求日志筛选
- 浏览器验收与回归自动化补强

目标：

- 让外部用户能持续升级和回滚，而不是只适合演示。

### P3：扩展能力

范围：

- 更多 CLI 深度接管
- 更完整的 Prompt / MCP / Workspace 联动自动化
- 更丰富的可观测性分析视图

说明：

- 这些能力重要，但不应阻塞 `v0.2` 主目标。

## 6. 数据模型与契约补强要求

后续实现必须继续坚持“数据与契约优先”，重点关注以下几类模型：

- `ProviderHealthState`
- `ProxyRequestLog`
- `HostIntegrationRecord`
- `McpHostSyncState`
- `PromptHostSyncState`
- 兼容性验收结果模型（建议新增）

建议：

- 兼容性验收结果不要散落在临时脚本里，至少要有稳定结构描述“CLI、场景、结果、失败原因、回归风险”。
- 健康状态与请求日志仍要保持“展示字段”和“敏感字段”分离。
- 所有新增模型先定义共享 schema，再落持久化和 API。

## 7. 风险分析

### 7.1 数据模型

- 如果继续只在运行逻辑里补边界，不同步沉淀稳定契约，后续导入导出、UI 和 CLI 会越来越难维护。

### 7.2 并发场景

- provider recovery probe、failover、host sync、prompt sync 都存在并发状态漂移风险。
- 后续所有增强都必须明确状态机边界，避免多个入口同时改写同一运行态。

### 7.3 故障模式

- 上游限流、证书问题、网络抖动、宿主机权限问题、systemd 用户会话异常，都是 Linux 无头场景里的常见故障。
- 这些故障必须通过“日志 + doctor + 推荐动作”形成闭环，而不是只返回一个错误码。

### 7.4 扩展性

- 当前单机 SQLite 架构足以支撑 `v0.2`。
- 在数据量放大 10 倍之前，不应引入不必要的分布式复杂度。
- 但所有运行态写入都必须继续维持“持久化存储 + 快照驱动数据面”的架构红线。

## 8. 推荐执行顺序

建议按以下顺序推进：

1. 为 `Codex` / `Claude Code` 补至少一轮真实验收记录
2. 补 `systemd --user` 日志查看、跟随和标准排障闭环
3. 收口路线图、交接文档和 README 的当前边界
4. 再补 Provider 编辑、导入导出和高级控制台体验

## 9. 文档维护要求

后续每完成一个 `P0` 或 `P1` 里程碑，至少同步更新：

- `README.md`
- `docs/linux-web-console-design.md`
- `docs/linux-operations-runbook.md`
- `docs/v0.2-requirements-plan.md`

如果实现已经改变产品边界，但上述文档未同步更新，则视为该里程碑未真正完成。
